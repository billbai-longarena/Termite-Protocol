# 放弃 Agent 记忆：当我们让 AI 像白蚁一样协作

> **"所有白蚁都是无状态的。环境承载了真正的智能。"**
>
> — 白蚁协议 v5.0 核心原则

---

现在的 AI Agent 框架，正在变得越来越重。

为了让大模型完成复杂的长线工程任务，我们给 Agent 塞进了跨会话记忆（Memory）、复杂的反思循环（Reflection）和庞大的状态机。结果是：Token 消耗呈指数级增长，上下文窗口被噪音和历史记录填满，一旦某个节点崩溃，整条 Agent 链路就像多米诺骨牌一样轰然倒塌。

我们在白蚁协议（Termite Protocol）的开发中，走了一条完全相反的路。

**假设所有 AI Agent 都是完全无状态的、即用即抛的——它们还能协作完成一个完整的软件项目吗？**

答案是：能。而且远比想象中更稳定、更具韧性。

在最近完成的五模型异构集群实验（A-006）中，我们见证了 Codex、Haiku、GPT-4o mini、Gemini Flash 和 Sonnet 五个不同能力量级的 AI，在完全无状态的状态下共同完成了一个 Python FastAPI + Go Gateway + React 全栈项目，历时 17 小时，产生了 **559 次 Commits、112 个完成信号**。

这背后，是白蚁协议在 v5.0 版本迎来的一次根本性的哲学转向。

---

## 一切从一道比喻题开始

白蚁协议的设计起点，是一道来自真实工程场景的比喻题：

**"乱石墙工人选石头的时候，需要判断石头的形状、感知相邻石头的缝隙、预估承重结构，每块石头都不一样，但最终能堆成坚固的城墙。白蚁协议驱动的 AI 们，能达到这个效果吗？"**

逐条拆解这道题，我们发现了六个盲区：

| 工人的能力 | 协议现状 |
|-----------|---------|
| 读石：判断材料形状 | 信号只有 type/weight，缺少"形状"描述 |
| 读缝：理解缺口形状 | HOLE 信号有，较好 |
| 试放：低成本探索 | claim 是二元锁，无轻量探测 |
| **邻石感知：依赖关系** | **信号之间无结构化依赖** |
| 承重判断：结构重要性 | weight = 紧急度，不等于结构重要性 |
| 墙体轮廓：全局结构 | .field-breath 只有标量摘要 |

这些盲区的存在，促使我们追问一个更深层的问题：**协议的基础隐喻——"白蚁是盲目的"——本身是否准确？**

---

## 洞察一：白蚁不是盲的，白蚁是无状态的

"白蚁是盲的"，在协议早期版本中，这是描述 AI Agent 工作模式的核心比喻。它推导出的设计哲学是：所有 Agent 平等、遵循相同规则、通过信息素进行间接协调。

但我们把"盲"这个字拆开，发现它实际上包含三层不同的含义：

| 层次 | 含义 | 对 LLM 是否成立 |
|------|------|----------------|
| 盲-1 | 无跨会话记忆 | **成立**——基础设施层面的事实 |
| 盲-2 | 只能感知局部上下文窗口 | **成立**——context window 有限 |
| 盲-3 | 不理解自己在建造什么 | **强模型不成立** |

问题恰恰出在盲-3。

我们回顾了所有实验数据：

- **A-003 ReactiveArmor**（纯弱模型实验）：2 个 Haiku 并行运行。基础协议循环（ARRIVE → SENSE → ACT → DEPOSIT）执行成功，但判断型行为全部失败——14 条观察中 9 条退化，规则涌现为零。**盲-3 成立。**
- **A-005 touchcli**（1 个强模型 + 弱模型群）：Codex 作为"牧羊人"，Haiku 负责执行。任务交接率 99%，观察质量 96.4%，项目交付成功。**强模型：盲-3 不成立。**
- **A-006 touchcli**（5 模型异构）：最高吞吐（559 Commits），但弱模型产出质量退化到 57%。**混合状态。**

结论非常清晰：**协议的每一个成功案例，都依赖于强模型"违反"了盲-3**。强模型看见了项目结构，把它的理解编码进信息素链，后继者才能在正确的节点继续工作。

因此，v5.0 用一个更精确的替代原则，取代了"白蚁是盲的"：

> **旧原则**：所有白蚁是盲的，通过信息素协调。
>
> **新原则（v5.0）**：所有白蚁是无状态的，环境承载智慧。
>
> - **原则 A（无状态，不可协商）**：任何白蚁可以在任何时候消失，不丢失系统性信息。所有持久知识必须存在于环境中。
> - **原则 B（环境承载智慧，设计杠杆）**：智慧存在于信号系统中，不在个体 Agent 脑中。强模型的核心职责是向环境注入结构化知识；弱模型的核心职责是从环境中读取指令并执行。

## 洞察二：11 个概念的重新澄清

确立了新原则之后，我们逐一重新审视了协议中的所有核心概念，发现许多概念存在误用或混淆。以下是最关键的几条：

**信息素（Pheromone）混淆了两种完全不同的东西：**

- **痕迹（Trace）**：工具保证的事实，如 git commit 记录。可靠性极高，不应衰减。
- **沉淀（Deposit）**：模型产出的知识，如观察和判断。可靠性取决于模型能力，需要质量加权和衰减机制。

把这两者混为一谈，导致我们对信号系统的信任级别设计出了偏差。

**Stigmergy（信息素协调）不是间接协调：**

自然界中，蚂蚁通过化学信号进行的是真正意义上的"间接"协调——信号改变环境，后来者响应环境变化。但在白蚁协议中，强模型写下 `observation_example` 时，**明确知道后继者会读到它，并且会根据它来调整自己的行为**。这是显式通信，而不是间接协调。这意味着：通信质量是一等设计变量，不是"反正信号留在那儿，随便吧"。

**牧羊人效应（Shepherd Effect）是核心机制，不是部署建议：**

在早期版本中，"推荐在集群中放一个强模型"只是一条经验性的部署建议，排在协议文档的很后面。但 A-003 的失败和 A-005 的成功共同证明：没有牧羊人效应，协议退化为机械循环。它应该在协议 Part I（核心机制）的地位，而不是 Part III（部署建议）的脚注。

## 洞察三：为什么"因材施教"是个陷阱

在面对强弱模型能力差异时，v4.0 的 PE-005 提案尝试了一个直觉上很合理的方案：在 Agent 到达时，通过历史产出质量进行能力测试，然后给强模型发"架构师工单"，给弱模型发"执行工单"。

这个方案在工程上行得通，但存在五个结构性缺陷：

| 缺陷 | 说明 |
|------|------|
| 能力是向量不是标量 | 同一模型可能代码强、架构弱；Python 强、Rust 弱 |
| 冷启动困难 | 前 3 次默认执行角色，可能持续浪费强模型 |
| 自我实现偏差 | 简化工单 → 简化产出 → 确认"果然是弱的" |
| 模型进化 | 新模型发布后，所有分类阈值失效 |
| 概念开销 | 3 套模板 + 推理算法 + DB 字段 + 测试逻辑 |

更深层的问题是：**"强"和"弱"是相对概念，而且在不同任务上会反转。** 一个在代码实现上表现一般的模型，可能在文档撰写或测试设计上表现出色。预先分类会系统性地压制这种跨域能力。

v5.0 用一个来自长期智慧的方案替代了这种短期效率的做法——

### 仁者见仁，智者见智

> 森林不测试你是鹰还是蚂蚁。同一片森林，鹰看到全景，蚂蚁看到脚下的土壤。

统一的 `.birth`（Agent 启动时读取的环境摘要），包含三层自然交织的信息：

```
第一层（表面）：任务 + 行为模板 + 安全网
  → 任何模型都能提取并执行

第二层（中层）：战略上下文 + 规则 + 信号关系
  → 中等以上模型读懂并调整行为
  → 弱模型读到但不理解，忽略（无害）

第三层（深层）：涌现提示 + 蚁丘相位 + 结构判断邀请
  → 只有强模型响应
  → 弱模型无视（无害）
```

层次不是标注为"初学者/专家"的独立文档，而是在自然行文中交织——任务描述中自然包含战略上下文，行为模板中自然包含结构判断的线索。

**A-003 实验提供了一个反直觉的证据**：弱模型（Haiku）面对完整的协议文本时，**并没有因此崩溃或产生混乱**。它们忽略了不理解的部分，只执行了能执行的。观察退化是能力不足导致的，不是信息过载导致的。

这个发现确立了一个重要规律：**弱模型在信息过载时 → 忽略多余的（无害）。弱模型在信息不足时 → 无法获取所需的（有害）。** 因此，提供更丰富的上下文，比提供精简的上下文更安全。

---

## 然而：LLM 世界里没有"仁者见仁"的自然机制

承认这条：上面的原则是正确的方向，但有一个被忽视的前提需要被打破。

自然界里，鹰和蚂蚁能"仁者见仁"，是因为它们有**不同的感知器官**。鹰的视觉系统在硬件层面就过滤了土壤微观信息；蚂蚁的触角在硬件层面就不接收 500 米外的光学信号。过滤发生在感知之前——"看不到"的东西不会成为噪声，它根本不存在于计算中。

**LLM 没有这个机制。**

所有 Token 都经过 attention 计算。弱模型不理解的内容，变成的不是"不存在"，而是三种东西：

1. **噪声**：占用 attention budget 但不产生有效信息，稀释有用 Token 的权重
2. **Cargo Culting 素材**：A-006 的数据显示，弱模型从协议术语中"捡起"了词汇但错误使用。例如观察中出现了 `pattern: "HEARTBEAT-PHASE6-START"`，`detail: "0"` 这样格式正确、语义空洞的沉淀
3. **幻觉锚点**：不理解的概念成为生成时的干扰锚点，拉偏输出分布

这意味着，"仁者见仁智者见智"必须通过工程手段来实现，而不是期待 LLM 自然过滤。

### 我们发现的关键杠杆点

分析了三个方向的工程方案后，我们得出了一个意外的优先级结论：

**信号放大（进攻）的收益，远大于噪声过滤（防守）。**

数据如下：

| 实验 | 行为模板（behavioral_template） | 观察质量 |
|------|--------------------------------|---------|
| A-003（纯 Haiku，无牧羊人） | 无 | 35.7% |
| A-005（Codex 牧羊人 + Haiku） | 有（Codex 创建） | **96.4%** |
| A-006（5 模型异构） | 有，但被稀释 | 57.0% |

A-005 的关键在于：Haiku 看到的 `.birth` 是**未经过滤**的完整版本。但 Haiku 的观察质量达到了 96.4%。原因不是噪声被清除了，而是 `behavioral_template` 字段提供了一个极其具体的模仿靶标：

```yaml
behavioral_template:
  example: pattern="auth-error-handling" context="src/auth/" detail="found 3 files..."
```

弱模型看到这个就明白了：`pattern` 要什么样、`context` 是什么粒度、`detail` 需要多长多具体。**这不是过滤——这是给弱模型一个填空模板，让它知道"好"长什么样。**

结论：

> **正确的工程方案是两条路并行：**
>
> 1. **防守**：代谢循环定期清理低质量 deposit，防止信号环境腐化
> 2. **进攻**：强化 behavioral_template 的可见性和具体性，让信号路标强到不需要过滤也能被弱模型捕捉

## 五模型异构实验：最高吞吐与最深教训

A-006 是白蚁协议迄今规模最大、也是教训最深刻的实验。

**配置：**

| Agent | 工具 | 模型 | 角色 |
|-------|------|------|------|
| Agent 1 | Codex | Codex 5.3 | 牧羊人（强模型） |
| Agent 2 | Claude Code | Claude Haiku | 弱模型工人 |
| Agent 3 | OpenCode | GPT-4o mini | 弱模型工人 |
| Agent 4 | OpenCode | Gemini Flash | 弱模型工人 |
| Agent 5 | OpenCode | Claude Sonnet 4.6 | 中等模型工人 |

**核心指标：**

- 总 Commits：**559**（史上最高）
- 完成信号：**112 个**（S-001 → S-112）
- 任务交接质量：**100%**（137/141 次交接成功）
- 观察质量：**57%**（vs A-005 的 96.4%）
- 涌现规则：**21 条，全部退化**

21 条涌现出来的规则，全部是这种格式：

```yaml
trigger: "When I observe: heartbeat"
action: "Follow the pattern described in trigger"
hit_count: 0
```

触发条件是心跳信号 ID，动作是"执行触发条件描述的模式"——完全是一个循环定义，无法产生任何实际指导。这是弱模型从协议术语中"cargo cult"出来的产物：格式完全正确，语义完全为空。

A-006 给出了一个反直觉的洞察：**高吞吐不等于高质量。** 当弱模型占多数、牧羊人效应被稀释时，系统会产出大量格式正确但语义空洞的信号沉淀，污染后续 Agent 的决策环境。

559 次 Commits 的背后，是一堵"高度正确但结构脆弱"的石墙。

## 当前状态与未来方向

白蚁协议目前处于 v5.1。核心机制已验证，正在持续演化：

**已解决：**
- 信号依赖图（Parent-child decomposition）：父子任务关系，解决多 Agent 并发时的工作饥饿问题
- 信号质量评分（Artifact Quality Scoring）：每条沉淀有 0.0-1.0 质量分，涌现权重加权
- 规则质量门禁：拒绝触发条件只包含心跳/信号 ID 的退化规则
- 空闲蚁丘检测：所有信号完成后自动注入退出指引，防止心跳空转

**待验证：**
- behavioral_template 从单行文本改为多行结构化模板，是否能进一步提升弱模型质量
- 环境端过滤（layer-1 内容质量门禁）对弱模型产出的可测量影响
- A-006 的质量退化，究竟有多少来自平台兼容性问题，有多少来自信噪比问题

**长期方向：**

协议 v6.0 的方向，是将"环境承载智慧"从原则推进到实现：让 `.birth` 的内容由**蚁丘当前的状态**驱动，而不是由预设的模板驱动。一个处于早期创世阶段的蚁丘，和一个处于维护期的蚁丘，给出的环境摘要应该完全不同——不是因为 Agent 不同，而是因为**世界不同**。

## 加入这场实验

白蚁协议不是一个传统意义上的软件框架。它是一套**基于纯文本和 Bash 的、极低开销的跨会话协作规范**。

没有后台常驻进程。没有复杂的向量数据库。没有 HTTP API。一切都在你的 Git 仓库和文件系统中发生——`signals/`、`.pheromone`、`.birth`。任何能读写文件的 AI，都能加入这个蚁丘。

一行命令，在你的项目中建立蚁丘：

```bash
curl -sSL https://raw.githubusercontent.com/billbai-longarena/Termite-Protocol/main/install.sh | bash
```

如果你厌倦了越来越臃肿的 Agent 框架——如果你也相信**极简的无状态个体 + 结构化的智能环境**才是 AI 协作的终极形态——欢迎来到白蚁的巢穴。

**GitHub**: [https://github.com/billbai-longarena/Termite-Protocol](https://github.com/billbai-longarena/Termite-Protocol)

期待你的宿主项目，为白蚁协议带来新的进化突变。

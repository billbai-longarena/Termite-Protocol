# 为什么无状态 AI Agent 需要“环境”，而不是“对话”

## 副标题

一篇围绕环境优先协作、跨会话记忆与 Shepherd Effect 的实证型长文草稿。

## 正文草稿

所有 AI 编码 Agent 都有一个隐蔽但根本的问题：

**会话一结束，上下文就跟着死掉。**

这听起来像一句常识，但它造成的结构性错位，比很多工具真正承认的更深。
软件项目是持续系统，AI Agent 会话不是。

这也是为什么很多 agent workflow 在 demo 里很惊艳，到了真实仓库里却显得脆弱。
你当然可以让一个 agent 在单次会话中很聪明，但真正难的是：如何让一串彼此独立的会话，表现得像一个连续的工程过程。

很多系统试图用“更多对话”来解决这个问题。
一个 agent 把上下文交给另一个，另一个再总结，监督者维护角色历史，planner 维护线程。

只要所有参与者都足够强，这种方法是能跑起来的。
但一旦出现以下条件，它就会迅速变脆：

- 上下文窗口昂贵
- 有弱模型参与
- 工作跨越很多会话
- 多个 agent 长时间触碰同一个仓库

这正是 **白蚁协议 Termite Protocol** 想解决的问题。

它的核心论点非常简单：

**如果 Agent 天生是无状态的，那么协调能力就应该放在环境里，而不是放在 Agent 自身。**

白蚁协议不试图通过越来越长的聊天历史来保存连续性，而是把运行态写进仓库现场：

- 信号存放在 SQLite
- 任务认领通过原子事务完成
- 观察像信息素一样跨会话沉积
- 每个新会话读取一个压缩后的 `.birth` 快照

这样一来，连续性的负担就不再完全压在“当前这次会话”上。
环境承载智能，Agent 只负责感知和执行。

## 为什么“对话优先”的协调方式会失效

对话是一个很自然的默认值，因为它看起来像人类协作。
我们通过讨论协调事情，所以让 AI Agent 也通过讨论协调，好像理所当然。

但对无状态 Agent 来说，对话有几个结构性问题。

第一，它贵。
如果每个新工人都必须读角色定义、总结、历史对话和监督记录，那么在真正碰任务之前，你已经消耗了大量上下文。

第二，它脆。
如果交接完全依赖语言，那么每个 Agent 都必须通过转述来保留细节、意图和项目状态。
漂移几乎不可避免。

第三，它会惩罚弱模型。
一个弱模型未必不能沿着好的局部模式继续做事，但它通常不擅长在多轮对话里从零构造并稳定维持这种模式。

这正是很多多 Agent 系统悄悄失效的地方。
它们默认所有 Agent 都同样擅长思考、解释、记忆和协商。
现实并不是这样。模型强度不均衡，成本也很敏感。

## “环境优先”真正改变了什么

环境优先协议会换一个问题来问：

**一个全新的会话，最少需要读到什么，才能立刻继续有效工作？**

在白蚁协议里，答案不是“所有历史”。
而是“当前运行快照”。

这个快照就是 `.birth`。
新 Agent 到达，读取有上界的状态信息，看到当前蚁丘情境，发现最高优先级且未被认领的任务，然后开始工作。

这件事很重要，因为它改变了系统形状：

- 协调能力跨越会话边界保留下来
- 上下文成本从不断增长变成有上界
- 任务认领从“靠沟通”变成“靠机制”
- 场记忆从短暂的变成可累积的

仓库开始更像一个持续运作的工地，而不是一段越来越长的聊天记录。

## Shepherd Effect（牧羊效应）

到目前为止，这个项目最重要的发现，是所谓的 **Shepherd Effect**。

弱模型的问题，并不总是“完全做不出有用工作”。
更常见的问题是，它**无法稳定地主动发起高质量工作模式**。

这个区别非常关键。

在一次审计实验里，强模型先工作，在环境中留下了高质量范式。
后续弱模型通过 `.birth` 读到这个例子后，开始模仿它的结构。
结果是输出质量显著提高。

它的含义并不是“弱模型突然变聪明了”。
更实际的含义是：

**强模型可以先为环境播种，弱模型再高效率地利用这种结构。**

这使得“强弱混编蚁群”比两种常见极端都更值得研究：

- 所有工作都交给一个昂贵的强模型
- 让很多便宜模型彼此独立工作，然后任由质量退化

环境成了“模式传播介质”。
这比要求每个工人都具备完整的对话协调能力，更有杠杆。

## 这套协议真正适合什么

白蚁协议不是单 Agent workflow 的普遍替代品。
它最适合的是：仓库本身需要保留足够结构，让后来 Agent 不必重新发现一切，也能继续推进工作。

例如：

- 多 Agent 并行开发
- 长周期仓库
- 强弱模型混编团队
- 审计要求高的工程环境
- 由多个可拆分任务组成的大型重构

但它并不适合很小的任务、一次性脚本或纯探索型思考。
一个好的协议，必须明确说明自己**不适合什么**。

## 这件事为什么不只是一个仓库的问题

更深的问题并不是“某个协议是不是更好”。
而是：我们是否真的按今天模型的真实约束来设计 agent 系统？

今天的 Agent 还不是稳定持久、具备连续记忆的心智。
它们更像强大但间歇出现的工人。
如果现实是这样，那么连续性必须来自别处。

你可以继续用更复杂的 prompt 和 orchestration layer 去掩盖这个事实。
也可以承认 statelessness 是现实，并把连续性转移到 Agent 所处的环境中。

这就是白蚁协议的赌注。

不是说 Agent 会 magically 停止遗忘。
而是说：**当场本身记住了足够多的东西时，遗忘不再那么致命。**

跑一遍 smoke test。
读一读审计材料。
质疑它的假设。

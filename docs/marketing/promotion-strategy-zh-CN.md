# 推广战略与执行计划（中文）

## 1. 目标先定义清楚

白蚁协议不适合走“大众 AI 工具”路线。

它更适合一条窄而深的增长路径：

- 先拿下 **会长期碰真实仓库、已经被上下文断裂折磨过的人**
- 再把这些人转化为 **实验者、讨论者、案例提供者**
- 最后才去放大为 **更广泛的 Agent 工程方法论项目**

建议把推广目标拆成三个阶段：

- **认知**：让目标用户在 30 秒内理解“这是解决 stateless agents 连续性问题的协议”
- **验证**：让目标用户在 10 分钟内完成一次 smoke test，并看到 `.birth`、signal、claim 这些核心机制
- **转化**：让目标用户愿意留下 issue、discussion、审计反馈，或者把协议装进自己的仓库试跑

## 2. 一句话定位

> 白蚁协议不是让 Agent 通过对话协作，而是让无状态 AI 编码 Agent 通过环境协作。

对外表达时，始终保持这三个顺序：

1. **问题**：会话结束后，Agent 上下文断裂
2. **机制**：把协调能力放进环境，而不是聊天记录
3. **证据**：真实蚁丘、审计实验、Shepherd Effect、60 秒 smoke test

不要一上来讲“多 Agent 编排框架”。
那会把项目拖回一个过度竞争、且不利于差异化的分类里。

## 3. 核心受众

### 第一优先级：重度 AI Coding 用户

画像：

- 已经高频使用 Claude Code、Codex、Cursor、Aider 一类工具
- 在真实仓库里感受过“新会话像失忆”
- 对 workflow、prompt 体系、仓库自动化有改造意愿

他们最关心：

- 能不能减少重复解释项目背景
- 能不能让多个会话稳定接力
- 能不能快速试，不需要先接入重系统

### 第二优先级：多 Agent / Agent Infra 构建者

画像：

- 正在做 agent orchestration、coding workflow、自动化调度
- 对 shared state、memory、handoff、auditability 有强关注

他们最关心：

- 这是不是一种可组合的底层协议，而不是封闭产品
- 是否有实验材料证明机制不是空想
- 能否作为自己的调度层或产品底座

### 第三优先级：工程团队中的方法论推动者

画像：

- CTO、技术负责人、平台工程师、研发效率团队
- 不一定亲自天天跑 Agent，但关心可靠性、审计和可复制流程

他们最关心：

- 是否适合长周期仓库
- 是否有清晰边界，不会把小任务复杂化
- 是否能形成团队规范和复用资产

## 4. 信息架构

### 核心主张

- **主张 1**：stateless agents 的核心问题不是不够聪明，而是连续性丢失
- **主张 2**：连续性应该由环境承载，而不是由不断增长的对话承载
- **主张 3**：强模型可以给弱模型播种，环境可以放大这种杠杆

### 三层内容结构

#### 第一层：钩子内容

目标是让别人停下来。

适合的话题：

- 为什么很多 AI coding workflow 一进真实仓库就断档
- 为什么“更多对话”不是多 Agent 协作的长期解
- 为什么弱模型不是不能用，而是缺少范式

#### 第二层：机制内容

目标是让别人理解“怎么做到的”。

适合的话题：

- `.birth` 为什么能替代重读整份协议
- SQLite signal + atomic claim 为什么比口头协调更稳定
- pheromone / observation / rule promotion 是怎么沉积场记忆的

#### 第三层：证据内容

目标是让别人愿意试。

适合的话题：

- A-005 的 Shepherd Effect
- 6 个真实蚁丘和 4 次审计实验
- 60 秒 smoke test
- audit package 怎么看、怎么复现实验路径

## 5. 渠道策略

### GitHub：主阵地

用途：

- 承接所有外部流量
- 完成首轮理解、验证和转化

重点动作：

- README 首页始终保持“问题 → 机制 → 证据 → smoke test”的顺序
- Release 页面每次只强调一个可复现改进点
- Discussions 用来承接“机制讨论”和“试跑反馈”
- Issues 用来承接安装、兼容性、脚本鲁棒性问题

### X / Twitter：打观点与实验结论

适合发：

- 简短 thesis
- 单张结构图
- 单个实验结论
- 新 release 的一个最强变化点

不要发成“功能列表”。
要发“一个让人愿意点进 GitHub 的判断”。

### 知乎 / 公众号 / 博客 / Medium / Substack：打长文

适合发：

- 为什么 stateless agents 需要环境，而不是对话
- Shepherd Effect 的实验和启发
- 在真实仓库中如何控制多 Agent 质量退化

这些渠道的任务不是带来大量泛流量，而是建立方法论权威感。

### Hacker News / Reddit / V2EX / 技术论坛：打讨论

适合切口：

- Show HN：开源协议 + 真实实验数据
- Ask HN：多 Agent coding 在长期仓库里如何保持连续性
- 经验贴：弱模型为什么在 shared environment 里会变得更可用

注意：

- 论坛首帖不要太像营销文
- 要把“哪里可能不成立”也主动讲出来
- 讨论比转化更重要，先拿高质量反馈

### 微信群 / Discord / Slack 小圈层：打试用

目标不是曝光，而是拿第一批高质量试跑者。

适合投放给：

- AI coding 重度用户群
- 开源维护者群
- Agent workflow 研究群

发法建议：

- 不发长海报
- 直接给一句判断 + 一张图 + smoke test 链接

## 6. 90 天执行计划

### 第 1 阶段：把入口打磨到能转化（第 1-2 周）

目标：

- 让冷启动访客能在仓库内完成理解和首轮验证

动作：

- 对齐 README、中英文首页文案、Quickstart、Release Notes 的表述
- 把 smoke test 结果截图做成固定素材
- 准备一页“别人第一次来最该看的 3 个链接”
- 在 Discussions 里预设 3 个种子话题

完成标准：

- 新访客 30 秒能说出项目解决什么问题
- 新访客 10 分钟能完成一次跑通

### 第 2 阶段：用一个核心 thesis 打开认知（第 3-5 周）

目标：

- 建立“环境优先协作”这个认知标签

动作：

- 发布 1 篇中文长文
- 发布 1 篇英文长文
- 同步 3-5 条短帖，分别切不同角度
- 每条内容都统一回指 GitHub 上的 smoke test 和 audit evidence

推荐内容顺序：

1. 为什么 stateless agents 需要环境
2. Shepherd Effect 说明了什么
3. 为什么这不是又一个 orchestration 框架

完成标准：

- 开始出现围绕“环境 vs 对话协调”的讨论，而不是只问安装方法

### 第 3 阶段：拿第一批案例与反馈（第 6-9 周）

目标：

- 拿到真实外部试跑证据

动作：

- 邀请 5-10 个重度用户跑 smoke test 或装进真实仓库
- 收集安装失败点、理解断点、命名困惑点
- 选 2-3 个典型反馈做成改进日志
- 把“用户试跑案例”写成短案例帖

完成标准：

- 至少有 3 个外部试跑反馈
- 至少有 1 个可公开引用的使用案例

### 第 4 阶段：把项目从“概念”推到“方法论资产”（第 10-12 周）

目标：

- 让项目具备持续扩散能力，而不是靠一次 launch

动作：

- 整理 FAQ：和多 Agent 框架、memory 系统、workflow 工具的差异
- 输出一页 adoption guide：什么团队适合用，什么团队不适合
- 输出一页 benchmark / evidence index：实验材料怎么读
- 形成“每个 release 如何对外讲”的固定模板

完成标准：

- 外部讨论开始引用你的术语，而不只是引用项目名

## 7. 内容排期建议

每周建议保持这个节奏：

- **1 条短帖**：打一个判断或一个数据点
- **1 条机制帖**：解释一个核心机制
- **每两周 1 篇长文**：建立方法论叙事
- **每个 release 1 条复盘帖**：讲一个具体改进和一个具体证据

内容选题池可以固定成 8 个：

1. 会话失忆为什么是 AI coding 的结构性问题
2. 为什么 shared state 比 shared chat 更适合长期协作
3. `.birth` 为什么重要
4. SQLite signal + atomic claim 的工程意义
5. Shepherd Effect 的实验过程和边界
6. 强弱模型混编如何避免质量坍塌
7. 什么场景不适合白蚁协议
8. 审计包如何帮助方法论落地

## 8. 转化路径设计

不要把 CTA 设计得太散。
建议固定成四种：

- **最低门槛 CTA**：跑 60 秒 smoke test
- **讨论型 CTA**：来 Discussions 质疑机制
- **贡献型 CTA**：认领 onboarding / docs / shell robustness 相关任务
- **试点型 CTA**：把协议装进一个真实仓库试跑

推荐转化路径：

1. 外部内容看到观点
2. 点进 GitHub 看 README
3. 直接跑 Quickstart
4. 去 Discussions 留下反馈
5. 进入 release / audit / knowledge base 深读

## 9. 指标怎么盯

### 北极星指标

- **有效试跑数**：完成 smoke test 并留下反馈的人数

这是比 star 更重要的指标。

### 辅助指标

- GitHub unique visitors
- README 到 Quickstart 的点击率
- Release 页面浏览和下载
- Discussions 新增数量
- 安装失败反馈数
- 外部内容带来的 referrer 来源
- 首次反馈到 maintainer 响应时间

### 定性信号

- 别人是否开始复述“环境优先协作”这套说法
- 别人是否引用 Shepherd Effect 讨论强弱模型混编
- 别人是否开始主动拿自己的 workflow 来和你对比

## 10. 风险与应对

### 风险 1：被误解成“又一个多 Agent 框架”

应对：

- 所有文案先讲问题，再讲环境机制
- 少讲 orchestration，多讲 continuity 和 shared state

### 风险 2：概念太强，落地感不够

应对：

- 每次外宣都带 smoke test
- 每次长文都挂具体 audit / repo / command

### 风险 3：讨论很多，试跑很少

应对：

- 缩短首轮验证路径
- 所有渠道统一 CTA 到 Quickstart
- 优先修 onboarding 阻塞点，而不是继续扩素材

### 风险 4：术语门槛偏高

应对：

- 对外首层文案减少 pheromone / colony 一类术语密度
- 在 deeper docs 再展开协议内部语言

## 11. 现阶段最值得先做的 5 件事

1. 把这个项目的对外主标题固定成“无状态 AI Agent 的环境协作协议”
2. 以 Shepherd Effect 为中心，做一篇中英文都能传播的代表作
3. 每次外发内容都只讲一个判断，不讲完整功能表
4. 重点拿第一批外部试跑案例，而不是先追求大曝光
5. 以“有效试跑数”和“高质量讨论数”作为前 90 天核心指标

## 12. 一版最简执行口径

如果你只想记一版最短的推广策略，可以记这四句：

- **先讲痛点**：stateless agents 会失去连续性
- **再讲机制**：白蚁协议把协作放进环境
- **再讲证据**：真实蚁丘、审计实验、Shepherd Effect
- **最后只推一个动作**：先跑 smoke test

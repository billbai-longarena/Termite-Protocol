# Community Growth Playbook

This document turns the repository growth plan into an operating checklist for maintainers.

## Core goals

- Help a new visitor understand the project in under 30 seconds.
- Get a motivated user to a successful first run in under 10 minutes.
- Convert curiosity into feedback, discussions, issues, and repeat usage.

## Weekly metrics to review

### Discoverability

- GitHub Traffic page views
- Unique visitors
- Referring sites
- Release page views and downloads

### Activation

- New discussions opened
- New issues opened
- Install/onboarding problems reported
- Time from first visit to first successful smoke test, when users report it

### Community health

- First maintainer response time for issues and discussions
- Number of `good first issue` and `help wanted` tasks available
- Pull request review turnaround

## Monthly review questions

- Which channels drove the most qualified visitors?
- Which README section caused confusion most often?
- Which installation step fails most often?
- Which use cases convert from interest to adoption?

## Launch checklist

- README hero, quick demo, and proof section are current.
- `README.zh-CN.md` matches the English positioning.
- Latest release notes and `CHANGELOG.md` are up to date.
- Discussions are enabled.
- Issue templates route questions away from bug trackers.
- At least 5 labeled onboarding-friendly tasks exist.

## Channel strategy

- Publish one concise launch post with the core thesis.
- Publish one deeper write-up explaining the Shepherd Effect and experiment design.
- Re-share after each release with one concrete improvement and one reproducible example.

## Positioning reminders

- Lead with the problem: stateless agents lose context.
- Then show the mechanism: environment-based coordination.
- Then show the evidence: audits, commits, and quality deltas.
- Keep Commander positioned as optional automation, not the core protocol.

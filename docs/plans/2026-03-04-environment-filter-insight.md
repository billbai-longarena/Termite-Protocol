## 核心洞察："仁者见仁智者见智"在 LLM 世界不存在自然机理

### 来源

协议哲学修正文档 (`2026-03-03-protocol-philosophy-revision.md`) 批判 5：

> 森林里鹰和蚂蚁用**不同的感知器官**自然过滤。LLM 全用同一个 text token processor。弱模型没有"不同的感官"——每个 token 都进了 attention 计算，不理解的内容不是"不存在"，而是"噪声"。

### 为什么这个观点成立

自然界的"仁者见仁"有一个被忽略的前提：不同物种有不同的感知器官。鹰的视觉系统在硬件层面过滤了土壤微观信息；蚂蚁的触角在硬件层面不接收 500 米外的视觉信号。**过滤发生在感知之前**——"看不到"的东西不会成为噪声，它根本不存在于计算中。

LLM 没有这个机制。所有 token 都经过 attention 计算。弱模型不理解的内容变成三种东西：

1. **噪声**：占用 attention budget 但不产生有效信息，稀释有用 token 的权重
2. **Cargo culting 素材**：A-006 数据证明——`pattern: "HEARTBEAT-PHASE6-START"` 就是弱模型从协议术语中"捡起"了词汇但错误使用
3. **幻觉锚点**：不理解的概念不会消失，它们成为生成时的干扰锚点，拉偏输出分布

### 精炼表述

> **"仁者见仁智者见智"在 LLM 世界里不存在自然机理。存在的是"智者提取信号，弱者被噪声淹没"。**

### 对协议的影响

哲学修正文档洞察 3 的结论（"原则成立，机制需修正"）需要进一步修正：

- **旧**：仁者见仁智者见智 → 统一文本，各自提取
- **问题**：LLM 没有自然的感知过滤，"各自提取"对弱模型意味着"被噪声淹没"
- **修正方向**：统一文本原则保留（不分类 agent），但**环境必须承担过滤职责**

比喻修正：不是让每只白蚁自己长出眼睛，而是**给蚁丘装上净水系统——所有白蚁喝到的水都是干净的**。

与 PE-005 的区别：
- PE-005：给不同白蚁喝不同的水（agent 分类）
- v5.0 原版：给所有白蚁喝同一条河的水，期望自己过滤（统一文本）
- 修正版：**过滤水源，然后给所有白蚁喝同一份干净的水**（环境端过滤）

## 工程方向分析

### 三层过滤器方案

| 层 | 做什么 | 实施点 | 可行性 | 可靠性 | 杠杆 |
|---|--------|--------|--------|--------|------|
| 1. 内容过滤 | .birth 构建时按内容质量筛选 | field-arrive.sh | 高 | 高 | 低-中 |
| 2. 信号自净 | 代谢循环归档低质量 deposit | field-cycle.sh | 高 | 中 | 中 |
| 3. 格式降噪 | .birth 用更结构化格式减少解析开销 | field-arrive.sh | 低 | 不确定 | 不确定 |

### 层 1 详情：field-arrive.sh 内容过滤

**已有基础设施**：
- `db_obs_best_example` 已按 quality_score 排序选最优 observation
- near-threshold clusters 的 SQL 已排除 `quality != 'low'`
- rule selection 有评分逻辑（hit_count + recency）

**需要做的**：
1. 规则质量门禁：Step 5 的 rule selection 跳过 trigger 是退化模式的规则（~5 行改动）
2. 信号摘要质量过滤：收益不大——信号本身由人类或强模型创建，通常不退化

**结论**：改动小（~15 行），风险低，收益明确。

### 层 2 详情：代谢循环归档低质量 deposit

**当前状态**：field-deposit.sh 已计算 quality_score 但 < 0.3 仍写入活跃集。

**关键设计决策**：过滤放在 field-cycle.sh（代谢循环）而非 field-deposit.sh（入口）。原因：
- deposit 接口保持简单，agent 调用后行为不变
- 过滤逻辑集中在代谢循环，与 decay/drain 逻辑同层
- 避免弱模型因 deposit 被"拒绝"而困惑

**误判风险**：H2 验证准确率 99.6%。A-006 的 270 条中 ~1 条好 observation 被错误归档。可接受。

**涌现影响**：quality_score < 0.3 的 observation 对涌现几乎无贡献（0.3 + 0.65 + 0.85 = 1.8 < 3.0 无论有没有那个 0.3）。理论风险，实际影响极小。

### 层 3 详情：格式降噪

**结论**：不做。没有实证数据，无法验证，投入产出比不合理。

## 关键发现：信号放大 > 噪声过滤

### 数据支撑

| 实验 | behavioral_template | observation 质量 |
|------|---------------------|-----------------|
| A-003 (纯 Haiku) | 无 | 35.7% |
| A-005 (Codex+Haiku) | 有 (Codex 创建) | **96.4%** |
| A-006 (5 模型) | 有，但被稀释 | 57.0% |

A-005 的 Haiku 和 Codex 看到的是**同一份未过滤的 .birth**。但 Haiku 达到了 96.4% 的 observation 质量。原因不是噪声被过滤了，而是 behavioral_template 提供了一个足够具体的模仿靶标。

### 机理分析

**噪声过滤**（防守性）：
- 删掉退化规则、归档低质量 observation
- 防止环境恶化
- 收益有天花板——即使删掉所有噪声，弱模型还是需要知道"做什么、怎么做"

**信号放大**（进攻性）：
- 让 behavioral_template 更突出、更具体、更容易模仿
- 直接提升弱模型产出质量
- A-005 证明：一个好的行为模板足以让 Haiku 从 35.7% 跳到 96.4%

### 结论

> **正确的工程方案是两个方向并行：**
> 1. **防守**：代谢循环定期清理低质量 deposit，防止路标被垃圾覆盖
> 2. **进攻**：强化 behavioral_template 的可见性和具体性，让路标本身更醒目
>
> 既然模型没有自然过滤机制，那就**让环境自带过滤（防守）+ 让信号强到不需要过滤也能被捕捉（进攻）**。

### behavioral_template 当前实现

field-arrive.sh 中 behavioral_template 在 .birth 的 `## task` 段末尾（lines 584-588），一行文本：
```
behavioral_template:
  example: pattern="auth-error-handling" context="src/auth/" detail="found 3 files..."
```

弱模型看到这个就知道：deposit 时 pattern 要什么样、context 什么粒度、detail 需要多长多具体。**这不是过滤，这是给弱模型一个"填空模板"。**

A-006 退化的主因可能不是 .birth 噪声太多，而是 3 个 OpenCode agent 可能没正确读取 behavioral_template（平台兼容性问题）。

## 待验证假设

1. **层 1 过滤是否对弱模型有可测量的差异？** — 需要 A/B 实验：同一弱模型，一组有退化规则的 .birth vs 一组过滤后的 .birth
2. **层 2 代谢归档是否改善涌现质量？** — 需要在有大量 observation 的蚁丘上测试涌现前后对比
3. **behavioral_template 的改进空间有多大？** — 当前是单行文本，是否改为多行结构化模板能进一步提升弱模型质量
4. **A-006 退化是平台兼容性问题还是信噪比问题？** — 需要隔离变量：同一平台（Claude Code）下，多弱模型 + 1 强模型，看质量是否回到 A-005 水平
